## Features

### Meeting File

One-line summary: One file per meeting with two sections — Prep (auto-generated before) and Notes (user jots during) — processed after the meeting to update the vault.

- **No meeting registry required.** Meetings read from calendar MCP. Meeting type inferred from multiple signals:
  - Attendee count: 2 attendees = likely 1:1
  - Calendar event title: project names from registry matched against title (e.g., "Auth Migration Status Review" → project meeting for Auth Migration)
  - Attendee composition: all your directs = team meeting, cross-team mix = project/coordination meeting
  - Recurrence: recurring = established meeting, one-off = adhoc
  - When inference isn't confident, ask the user on first encounter. Once classified, remember it for future occurrences (stored as optional override in registry).
  - Optional manual overrides in registry for meetings that need special config (project association, custom debrief type, name alias).
- File organization by inferred type:
  - `Meetings/1-1s/{person-name}.md` — one file per person, sessions appended chronologically
  - `Meetings/Recurring/{meeting-name}.md` — team meetings, standups, syncs
  - `Meetings/Adhoc/{meeting-name}.md` — one-off meetings
- **Meeting file has two sections:**

#### Section 1: Prep (auto-generated before the meeting)
- Generated by morning sync or on-demand:
  - "Prep for my 1:1 with Sarah" → single meeting
  - "Prep for my remaining meetings" → all unprepped meetings left today
  - "Update prep for my meetings" → reads existing prep (including any user-added topics) as context, then appends only what's new since last prep (new emails, completed tasks, new blockers). Existing items are never removed or modified. User-added topics are accounted for — if the user already added a topic the agent would have surfaced, it's not duplicated.
- Every item is a checkbox — user checks off what was discussed during the meeting
- Content varies by meeting type:
  - **All meetings:** open action items between you and attendees, carry-forward items from last session (unchecked items from previous prep), recent decisions on relevant projects, open blockers
  - **1:1s — deeper prep because these are the highest-leverage meetings:**
    - Did YOU follow through on your action items? (easy to forget — this prevents the embarrassment)
    - Recent contributions and project work since last 1:1 (factual list from project timelines and task completions)
    - Pending feedback with coaching-tone talking points (from observations log)
    - Career development context: growth areas in person file, recent observations tagged in those areas, time since last career conversation
    - Personal notes ("asked about the marathon last time")
  - **Project meetings:** open tasks for that project, recent timeline entries, dependency status, blockers, decisions needed
  - **Team standup/sync:** your updates to share (tasks completed, in progress), overdue delegations to follow up on, blockers across the team, action items from last standup
  - **Design/doc review:** link to the document being reviewed, related project context, previous decisions on this topic, pre-read prep if available (see writing-and-drafts Pre-Read Preparation)
  - **Cross-team meeting:** open dependencies on the other team, what you're waiting on from them, recent comms with attendees, what you need from this meeting
  - **Conversation coaching for sensitive items:** for prep items that involve difficult or high-stakes topics, the agent adds a brief suggestion on how to bring it up — calibrated to your communication style (from `communication-style.md`) and your relationship with the attendees (audience tier from registry). Examples:
    - Pending feedback → coaching-tone talking points ("Consider opening with the positive impact before discussing the growth area")
    - Critically blocked project → suggested framing for escalation ("Frame as impact to timeline, not blame — here's a suggested opening...")
    - Difficult conversation → suggested approach based on relationship tier ("With a peer, lead with shared goals. With upward, lead with data.")
    - Overdue delegation → how to follow up without micromanaging ("Ask about blockers first, not status")
  - Not every prep item gets coaching — only items flagged as sensitive (feedback, blockers, overdue items, escalations). Straightforward items stay as plain checkboxes.
  - All data points are factual (dates, counts, explicit entries) — the judgment is yours, not the AI's (D018)
- "Add topic to standup: discuss deployment timeline" → appends checkbox to Prep section

#### Section 2: Notes (user writes during the meeting)
- Minimal structure — user jots rough notes without spending time formatting:
  - **Notes** — free-form discussion points
  - **Action Items** — who does what by when (rough is fine, agent cleans up during processing)
  - **Decisions** — what was decided
- This is a scratchpad, not a polished document. The agent handles formatting during processing.

### Process Meeting

One-line summary: "Process this meeting" or "process my meetings" reads the meeting file and updates the vault — closes items, creates tasks, updates timelines.

- Triggered by: "process this meeting", "process my meetings" (batch), or via Universal Done ("done with 1:1 with Sarah")
- Reads the entire meeting file (Prep + Notes) and:
  - **Checked prep items** → marks as resolved/discussed. Closes associated TODOs where applicable.
  - **Unchecked prep items** → carries forward to next meeting's Prep section automatically
  - **Action items (for you)** → task list
  - **Action items (for others)** → task with `type:: delegation`
  - **Decisions** → project timeline with Decision category (D014)
  - **Observations about people** → person file
  - **Blockers** → project timeline
  - **Recognition** → person file
  - **Your contributions** → contributions log
- All extracted entries use the provenance marker system: `[Auto]` if explicitly stated in notes (user wrote "Action: Sarah to draft the doc by Friday"), `[Inferred]` if agent interpreted from context, genuinely ambiguous items go to review queue.
- Multi-destination routing applies — one meeting can produce entries across multiple destinations (see non-functional)
- **Meeting-type-aware processing** — different meeting types emphasize different extractions:
  - **1:1:** heavier on observations, feedback, career topics, personal notes
  - **Standup/sync:** lighter touch, mostly blockers and status updates
  - **Design review / decision meeting:** focuses on decisions, alternatives rejected, risks, action items
  - **Operational review:** metrics discussed, trends, action items for owners
- Agent infers meeting type from calendar data or asks if unclear
- Batch mode: "process my meetings" processes all meeting files with unprocessed notes from today

### Meeting Summaries from Email

One-line summary: Zoom/Teams AI meeting summaries arriving via email are appended to the meeting file AND processed standalone — two independent paths, dedup catches overlap.

- Email processing detects meeting summaries (subject line patterns like "Meeting Summary", "AI Notes from", sender patterns from Zoom/Teams/etc.)
- **Path 1 — Append to meeting file:** raw content appended to the Notes section of the corresponding meeting file (matched by meeting name + date from calendar). Available for user review when they process the meeting file later.
- **Path 2 — Standalone processing:** the summary is also processed through the regular email extraction pipeline (action items for you → TODO, action items for others → follow-up delegation, decisions → project timeline, etc.). Items staged in review queues immediately.
- **Two paths are independent** — no race condition. If the user never processes the meeting file manually, nothing is missed (email processing already handled it). If they do process it later, near-duplicate detection (dedup layer 3) catches items already staged from the email path.
- User-created emails containing meeting notes (forwarded to a project folder or DraftReplies) follow the same dual-path pattern.

### Time Block Planning

One-line summary: Plan your day with time blocks and optionally create personal calendar events with three-layer safety.

- "Plan my day" → suggests time blocks based on priorities, tasks due, and free calendar slots
- "Reserve 2 hours on Thursday for the design doc" → finds available slots, shows options, you pick
- "Reserve 3 hours this week for deep work" → scans the week for best slots, shows options
- Creates personal calendar events (no attendees — ever) with three-layer protection (D003):
  1. Agent instruction rule: never add attendees, always use configured prefix
  2. Pre-tool check: rejects any call with attendees or missing prefix (where AI model supports hooks)
  3. Explicit confirmation: shows all event parameters before creating
- Calendar event prefix includes type: `[Myna:Focus]`, `[Myna:Task]`, `[Myna:Reminder]` — so you can tell at a glance what the block is for. Prefix and type labels configurable in workspace config.
- For AI models without hook support, layers 1 + 3 still protect (instruction + confirmation)

### Calendar Reminders

One-line summary: Create time-sensitive calendar reminders that push notifications via your calendar app.

- Two types:
  - **Task reminder:** "Remind me about the design review at 2pm" → creates calendar event linked to the task
  - **Standalone reminder:** "Remind me to call Alice at 3pm" → creates calendar event, no Obsidian task
- All calendar events use typed prefix: `[Myna:Task]`, `[Myna:Reminder]` — consistent with time block planning. Three-layer protection (D003), no attendees.
- Dashboard-level reminders (tasks with due dates shown in daily note) are handled by daily-workflow, not here

### Follow-Up Meeting Draft

One-line summary: Draft a follow-up meeting invite for the user to manually create in their calendar.

- **Two trigger paths:**
  - **Agent conversation:** "Draft follow-up meeting with Sarah and Alex about the auth migration" → creates draft
  - **DraftReplies folder:** reply to an email thread with instructions like "create follow-up meeting invite — include Sarah and Alex, discuss next steps on cache decision" → agent creates the draft as part of DraftReplies processing (see writing-and-drafts Email Draft Reply)
- Draft created in `Drafts/Meeting/`
- Includes: suggested attendees, proposed agenda, time suggestion based on attendee availability (if calendar MCP available), context from recent discussions and the email thread (if triggered from email)
- Auto-creates linked TODO for tracking ("send follow-up invite for auth migration")
- User manually creates the calendar invite from the draft — Myna never creates events with attendees (D003)
