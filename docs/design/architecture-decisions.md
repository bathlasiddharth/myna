### D065 — Content-first log entry format: category/content before date

**Date:** 2026-05-10
**Context:** Agent-written log entries (Timeline, Observations, Recognition, Tasks, Contributions) were formatted with a leading date: `[{YYYY-MM-DD}] **{category}:** {content}`. This buried the substantive content behind provenance metadata, making entries hard to scan and reinforcing a "log-first" mental model over a "content-first" one.
**Decision:** Log entries use content-first format: `- {content} [{provenance}] ({source-type}, {identity}, {YYYY-MM-DD})`. The date and source move to a trailing parenthetical. Personal notes written by the user (not agent-captured entries) retain whatever format the user chose — no reformat. Contributions Log format: `**{category}:** {description}; result: {result} [{provenance}] ({source}, {user.name}, {YYYY-MM-DD})`.
**Alternatives rejected:** Keep date prefix for sortability (log sections are newest-first already — the date in the trailing field is sufficient for reference). Keep markdown bold date (clutters the visible label). Separate date column (breaks plain-markdown readability).

### D064 — Newest-first ordering for all agent-written log sections

**Date:** 2026-05-10
**Context:** Several skills appended log entries to the bottom of sections (append-only, chronological order). This meant users scrolling to the bottom to find the latest data — opposite of how people read notes. Append-only was treated as a safety property but it conflated two things: immutability of existing entries (correct) and write direction (arbitrary).
**Decision:** All agent writes to log-like sections prepend new entries at the top (newest at top). This applies to: Timeline, Tasks, Observations, Recognition, Contributions Log, Source File, and meeting session blocks within meeting files. Exception: daily note `## End of Day` blocks are written at the bottom of the file (correct — they are created once and the daily note structure is top-to-bottom narrative). Existing entries are never modified or deleted — only new entries are prepended.
**Alternatives rejected:** Append (requires scrolling to bottom; inconsistent with how date-sorted lists are normally read). Sorted insert (requires parsing existing dates, fragile, violates tolerance).

### D063 — Wiki-link aliases required at entity file creation; readable form in all skill output

**Date:** 2026-05-10
**Context:** Skills were writing wiki-links using the slug form (`[[people/sarah-chen]]`, `[[Projects/auth-migration]]`). These render in Obsidian but are fragile: if the file moves or is renamed, links break. More importantly, they expose internal naming conventions in user-facing content when readable aliases should be used instead.
**Decision:** (1) Skills that create entity files (project, person, meeting) must set `aliases:` in the frontmatter at creation time. Minimum alias: the display name. Examples: `aliases: ["Sarah Chen"]` for a person file, `aliases: ["Auth Migration"]` for a project file, `aliases: ["Sarah Chen 1:1"]` for a 1:1 meeting file. (2) All skill-written wiki-links use the readable display form: `[[Sarah Chen]]`, `[[Auth Migration]]`, `[[Sarah Chen 1:1]]`. The alias in the frontmatter resolves this to the slug file. Slug-form links in existing vault files are not retroactively changed — tolerance applies.
**Alternatives rejected:** Always use slug form (hard to read, exposes internals). Always use full path form like `[[People/sarah-chen]]` (even harder to read, breaks on vault reorganization).

### D062 — Project files use `## Tasks` for raw task storage; `## Open Tasks` is a Dataview view only

**Date:** 2026-05-10
**Context:** Skills were writing tasks to `## Open Tasks` or a sub-section `### Agent-Added Tasks` inside it. The `## Open Tasks` section contains a Dataview query block that automatically renders matching tasks. Writing raw task items to this section would break the Dataview query or silently be ignored by Obsidian, depending on placement.
**Decision:** Project files have two task-related sections with distinct roles: `## Tasks` is the raw task storage — skills write all new tasks here using `- [ ]` syntax with `📅 YYYY-MM-DD` due dates. `## Open Tasks` is a Dataview live view section — it contains a Dataview query block, skills NEVER write inside or around it. The `### Agent-Added Tasks` sub-section is removed entirely. This split is documented in `docs/design/file-formats/entities.md § Project File`.
**Alternatives rejected:** Write tasks directly under `## Open Tasks` (corrupts or is silently ignored by the Dataview block). Create a new section per skill (proliferates sections, user has to track multiple places).

### D061 — Canonical daily note structure: Morning Focus, Sync blocks, End of Day

**Date:** 2026-05-10
**Context:** The original daily note template had `### Immediate Attention`, `### Capacity Check`, and `### Open Tasks` (a Dataview block) in the morning section. Four reader skills (wrap-up, plan, weekly-summary, sync) depended on `### Immediate Attention` to find actionable tasks for the day. This created an active-drift bug: `### Immediate Attention` was removed from the canonical template as the design evolved, but the reader skills still referenced it. Skills reading a non-existent section silently got no data.
**Decision:** The canonical daily note has two types of sections: (1) `## Morning Focus` — free-form text the user types as their intent for the day (not agent-populated). (2) `## Sync — {HH:MM}` blocks containing `### Briefing`, `### Today's Meetings`, `### Tasks Due Today`, `### Dashboards` — created by the sync skill. End-of-day section is `## End of Day — {HH:MM}` at the bottom, containing `### Planned vs Actual`, `### Contributions Detected`, `### Carried to Tomorrow`, `### Quick Notes`. Reader skills (wrap-up, plan, weekly-summary) must not read `### Immediate Attention` or any synthesized daily-note section for task data — they query source files (`Projects/`) directly.
**Alternatives rejected:** Restore `### Immediate Attention` (adds a Dataview block that needs maintenance, creates a second source of truth for task data). Keep old template and update skills to match (the old structure had confusing Dataview blocks in the daily note that didn't add value over querying project files directly).

### D060 — Read Principle: query source files for complete datasets, daily/weekly notes for user intent only

**Date:** 2026-05-10
**Context:** Several reader skills (wrap-up, plan, weekly-summary) read daily and weekly notes as their primary source of task and work data. This made skill output dependent on whether the daily note had been fully populated — a timing dependency. If sync hadn't run, or the daily note was from yesterday, skills got stale data.
**Decision:** Skills reading complete datasets (all open tasks, all observations about a person, all contributions this week) must query source files: `Projects/` for tasks, `People/` for observations/recognition, `Meetings/` for meeting history. Skills should read daily/weekly notes only for user-supplied context: `## Morning Focus` for today's intent, `## Quick Notes` for user-typed observations, direct user messages or annotations. This principle is documented in `docs/design/file-formats/_conventions.md ## Read Principle`.
**Alternatives rejected:** Synthesize everything into the daily note and read from there (creates timing dependency, duplicates source-of-truth, breaks when daily note is stale or absent). Read both always (redundant reads, deduplication complexity).

### D059 — Tolerance principle: schema describes what skills create, not what they require

**Date:** 2026-05-10
**Context:** Skills were brittle when user vaults diverged from the canonical template — for example, a section was renamed by the user, or a section the skill expected didn't exist because the user deleted it.
**Decision:** The file-format schema describes what skills create (section names, formats, entry structures). When reading, skills locate target content by section name first, then fall back to content patterns (provenance markers, task syntax, frontmatter fields). When writing, skills find the section by name and add to it; if the section is absent, create it in a sensible position. Skills never reorder, rename, or delete user content to "match the schema." This is documented in `docs/design/file-formats/_conventions.md ## Tolerance`.
**Alternatives rejected:** Fail loudly if schema sections are missing (too brittle, breaks for any customized vault). Ignore missing sections silently (loses data). Re-create the entire file from template (destroys user content).

### D058 — Skill schema references: on-demand reads from `~/.claude/myna/file-formats/`

**Date:** 2026-05-10
**Context:** `docs/design/foundations.md` §2 was the monolithic canonical file-format reference. As skills were built, some referenced it and some didn't — leading to format drift where skills wrote entries in inconsistent formats. The file was too large to read in full for every skill operation, and references to it were vague (no section anchors).
**Decision:** Each skill that reads or writes vault files must include a schema reference line specifying the exact file(s) and section anchors in `~/.claude/myna/file-formats/` that govern its vault operations. Skills read only the section(s) they need — not the entire schema directory. Reference format: `Before reading or writing {file type}, read ~/.claude/myna/file-formats/{file}.md, section ## {Section Name}.` Multiple files/sections are listed as a bulleted list. Skills read the schema on demand at the moment they are about to read/write — not at session start.
**Alternatives rejected:** Inline all format specs in each skill (duplication, maintenance burden — a format change requires updating every skill). Reference the full `foundations.md` (too large to scan, no anchors, leads to approximation). No schema reference (proved to cause drift — skills write in inconsistent formats without a live reference).

### D057 — Schema centralization: `docs/design/file-formats/` ships to `~/.claude/myna/file-formats/`

**Date:** 2026-05-10
**Context:** `docs/design/foundations.md` §2 contained all canonical file-format schemas (project files, person files, meeting files, journal, etc.) inline. This was the only authoritative reference for vault format, but skills couldn't reference it at runtime — the file was in the repo, not installed. Skills increasingly diverged from canonical formats as they were authored without a live reference.
**Decision:** File-format schemas are extracted into a dedicated directory `docs/design/file-formats/` (7 files: `_conventions.md`, `entities.md`, `meetings.md`, `journal.md`, `drafts.md`, `queues.md`, `system.md`). The install script copies this directory to `~/.claude/myna/file-formats/` so it is available at runtime. `foundations.md` §2 is narrowed to a rationale-only pointer — it explains *why* the schema exists and links to `file-formats/`, but does not duplicate the schemas inline. Skills reference schema files by file + section anchor on demand (see D058).
**Alternatives rejected:** Keep schemas in `foundations.md` and have skills embed format specs (duplication, drift). Ship schemas to the vault (not a system file — vault is user data space). Reference `foundations.md` at runtime (file is in the repo, not installed, and is too large to scan for a single section anchor).

### D056 — Journal/ rolling archive replaces archive_after_days config
**Date:** 2026-05-01
**Context:** The vault had a flat `Journal/` folder with a `journal.archive_after_days` config field (default: 30). This created two problems: (1) daily and weekly notes accumulated in `Journal/` root making it progressively cluttered, and (2) the archive threshold was an unnecessary config decision that users had to make upfront without context. In practice no user needs more than one active daily, weekly, or monthly note — the previous note becomes irrelevant the moment a new one is created.
**Decision:** `Journal/` root holds at most three files at any time — the current daily note (`{YYYY-MM-DD}.md`), current weekly note (`{YYYY-W\d\d}.md`), and current monthly note (`{YYYY-MM}.md`). When the sync skill creates a new note of any type, it first globs `Journal/*.md` for files matching the appropriate pattern that are not the new note being created, then moves any found file to `Journal/Archive/Daily/`, `Journal/Archive/Weekly/`, or `Journal/Archive/Monthly/` respectively using Bash `mv`. The `journal.archive_after_days` config field is removed entirely — no threshold, no user decision, no accumulation. Install creates `Journal/Archive/Daily/`, `Journal/Archive/Weekly/`, and `Journal/Archive/Monthly/` at setup time.
**Alternatives rejected:** Keep archive_after_days with a lower default (still accumulates N files before archiving, still requires a config decision, still clutters the folder). Single `Journal/Archive/` folder without type subfolders (harder to navigate archived notes; type subfolders make it trivial to find "all my past daily notes"). Manual archiving only (defeats the purpose of automation; users reliably forget and the folder fills up).

### D055 — Vault subfolder fixed to `myna`; no longer user-configurable during setup
**Date:** 2026-05-01
**Context:** D011 said the subfolder name is configurable during setup. In practice, asking for it adds friction for no real benefit — almost no user will want a name other than `myna`, and the option confused new users who don't yet have a mental model of the vault structure. Removing it simplifies Step 1 of `/myna:setup` to a single question.
**Decision:** The Myna subfolder is always `myna`. `/myna:setup` does not ask for it; the install script is called with `"myna"` hardcoded. Users who want a custom subfolder name can edit `~/.myna/config.yaml` directly after install and re-run the install script. Overrides D011's "configurable during setup" clause.
**Alternatives rejected:** Keep the question with `myna` as a default users can accept (still adds a prompt new users don't understand; the default is always taken in practice).

### D054 — Onboarding redesign: single `/myna:setup` entry point, shell-script scaffolding, Config UI primary path
**Date:** 2026-04-30
**Context:** The two-step first-time flow from D053 (`/myna:install` → `/myna:setup`) created unnecessary friction: users had to run two commands, and the split implied both steps were equally important milestones. The guided chat interview in `/myna:setup` added a third variation on top of Config UI and direct YAML editing, making the options paragraph harder to follow. Separately, the Config UI was being invoked from `~/.myna/ui/` — a path outside the plugin directory — which conflicted with where `server.py` actually lives after plugin install. And `~/.myna/install-manifest.json` (introduced in D049) was no longer needed once the plugin model (D053) took over version tracking and `server.py` began reading directly from `~/.myna/config.yaml`.
**Decision:** Five concrete changes land together as the onboarding redesign: (1) `/myna:setup` is the single entry point for all onboarding — new and returning users alike. It detects install state by reading `~/.myna/config.yaml` and runs the scaffolding step inline if config does not exist; `/myna:install` is retired. (2) Vault directory creation and `~/.myna/config.yaml` writing move to `scripts/install.sh` — a standalone shell script invoked by `/myna:setup` on first run. One `bash` call, one permission prompt, idempotent and re-runnable. (3) The guided chat interview is removed. After scaffolding, `/myna:setup` presents exactly two paths: Config UI (recommended) and doc import (alternative). Direct YAML editing remains available as a manual fallback, documented in the checklist. (4) The Config UI server (`server.py`) runs from the plugin directory, not `~/.myna/ui/`. `/myna:setup` references it via `${CLAUDE_SKILL_DIR}/../../ui/server.py`. (5) `~/.myna/install-manifest.json` is retired; `server.py` reads vault path and subfolder directly from `~/.myna/config.yaml`. Two pre-configured shell aliases (`myna` and `myna-ro`, with distinct tool permission sets) are offered at the end of every `/myna:setup` run — first-time and returning. D049's mention of `install-manifest.json` remains as historical record and is not edited.
**Alternatives rejected:** Keep two-command flow (`/myna:install` + `/myna:setup`) — the split adds steps without adding clarity; a single entry point that detects state is cleaner. Keep the guided chat interview as Option B — three input modes (UI, interview, YAML) is one too many; the interview duplicates what the Config UI does more ergonomically, and removing it simplifies the user-facing decision. Run `server.py` from `~/.myna/ui/` — that path doesn't exist after plugin install; the plugin directory is the authoritative location for bundled scripts. Keep `install-manifest.json` — it was introduced to track installed paths but `config.yaml` already holds the only two path values (`vault_path`, `subfolder`) that `server.py` needs; maintaining a second file adds complexity with no benefit.

### D053 — Plugin distribution via Claude Code plugin marketplace
**Date:** 2026-04-26
**Context:** The install-script model (D049) required users to clone the Myna repo and run `./install.sh` from the command line. This created friction: users needed git, a cloned repo that had to stay on disk for updates, and awareness of shell paths. The Claude Code plugin system provides a first-class distribution and update mechanism — installation is a single `/plugin install` command, updates are automatic, and the user never needs to touch a shell script.
**Decision:** Myna distributes as a Claude Code plugin instead of a standalone install script. Plugin name: `myna`. Install command: `/plugin install myna@agentflock` (after adding the agentflock marketplace via `/plugin marketplace add agentflock/plugins`). All skills live at `skills/*/SKILL.md` inside the plugin directory; the main agent lives at `agents/agent.md`. The skill namespace is `myna:` — skills are invoked as `/myna:{name}` (e.g., `/myna:sync`, `/myna:setup`), the agent is referenced as `myna:agent`. Vault path and subfolder are stored in `~/.myna/config.yaml`, written by `/myna:install` on first run. First-time setup flow: `/myna:install` (vault directory creation and config.yaml) → `/myna:setup` (guided identity, projects, people, communication style). Shell alias `alias myna="claude --agent myna:agent"` is optional but recommended for the `myna` shorthand. Vault config YAML files and `~/.myna/config.yaml` are never overwritten by plugin updates.
**Alternatives rejected:** Keep the install-script model from D049 (friction for new users — requires git clone, shell execution, and keeping the repo directory; no automatic updates). Hybrid of install script + plugin (two install paths to maintain). Manual copy of skill files to `~/.claude/skills/` (same friction as the install script without the update mechanism).

### D052 — Three-mechanism customization layer for update-safe skill overrides
**Date:** 2026-04-19
**Updated:** 2026-04-29 — Revised locations after D053 (plugin model). Files inside the plugin directory are overwritten on every plugin update, so per-skill overrides and routing overrides must live outside the plugin directory. Migrated to `~/.myna/overrides/`.
**Context:** Users who customize skills or add routing rules lose their changes when the plugin updates. The upstream SKILL.md files are always overwritten on plugin update, so any in-place edits inside the plugin directory are destroyed. A clean separation is needed between upstream content (overwritten on update) and user content (preserved). Per D053, `~/.myna/` is the stable user-owned location that plugin updates never touch.
**Decision:** Three mechanisms: (1) `~/.myna/overrides/skills/myna-{skill-name}.md` — per-skill override file. Users add overrides, extra steps, or behavioral tweaks. The agent reads this file at skill execution time (before the built-in SKILL.md content applies) via Claude Code's built-in Read tool; override content takes precedence when it conflicts with defaults. The global rule is wired once in `myna-steering-system` so no individual SKILL.md needs to reference it. Created by `/myna:init` if missing; never overwritten by plugin updates. (2) `~/.myna/overrides/routing.md` — routing rules for user-added skills. The agent reads it at routing resolution time if it exists; its rules take precedence over the built-in routing table. Created by `/myna:init` if missing; never overwritten. (3) User skill directories — named `myna-[prefix]-[name]` (e.g., `myna-amazon-oncall`). The single-word pattern `myna-[name]` is reserved for built-in skills. Plugin updates only touch built-in skill content; user-prefixed skill directories are never touched. Update behavior: built-in SKILL.md files inside the plugin directory are always overwritten on plugin update. Everything under `~/.myna/overrides/` and user skill directories are never touched.
**Alternatives rejected:** `CUSTOM.md` alongside `SKILL.md` inside the plugin directory (overwritten on every plugin update — incompatible with D053). `~/.myna/custom/` directory name (less self-documenting than `overrides/` — new users wouldn't immediately know what these files do). `~/.myna/custom-routing.md` at root (inconsistent with per-skill overrides living under `overrides/`; `overrides/routing.md` keeps all user overrides under one namespace). In-place editing of SKILL.md files (overwritten on update, changes lost). Separate branch/fork per user (high friction — users would need to merge upstream changes). Storing user customizations in the vault (vault is user data, not agent instructions; mixing them would be confusing).

### D051 — "Chief of Staff" framing replaces "personal assistant"
**Date:** 2026-04-14
**Context:** Myna was described as a "personal assistant" throughout docs, prompts, and agent files. The framing undersells the product — "personal assistant" implies a reactive, task-execution role, while Myna actively manages the information layer of the user's job, surfaces priorities, preps briefs, tracks people and projects, and keeps the user unblocked. "Chief of Staff" captures the scope and proactive nature of the role better.
**Decision:** All descriptions of what Myna is use "Chief of Staff for tech professionals" instead of "personal assistant for tech professionals." Product name (Myna) and runtime terminology ("AI agent", "Claude", "LLM") are unchanged.
**Alternatives rejected:** Keep "personal assistant" (accurate but undersells scope and proactivity). "Executive assistant" (connotations of calendar management and gatekeeping, not information management). "Second brain" (popular but implies pure storage, not active workflow management).

### D050 — 24 feature skills + 6 steering skills as native Claude Code SKILL.md files
**Date:** 2026-04-12
**Context:** D049 described an install model with 14 skills (flat `.md` files in `~/.myna/skills/`), 5 steering files inlined into the agent body, and an MCP server for vault operations. The Phase 2 rebuild (P2-T10 through P2-T14) rearchitected to use Claude Code's native skills mechanism: skills are `SKILL.md` files inside named directories under `~/.claude/skills/`, with frontmatter that Claude Code reads for auto-invocation matching. Steering rules became 6 proper skills (adding `myna-steering-memory` and `myna-steering-vault-ops`) preloaded via the agent's `skills:` frontmatter field instead of being inlined. The MCP server was removed entirely — vault operations use Claude Code's built-in tools (Read, Write, Edit, Grep, Glob) guided by the `myna-steering-vault-ops` skill. The original 14 skills were expanded to 24 to cover features that had been bundled (e.g., `brief` split into `brief-person` and `brief-project`, people-management features surfaced as `team-health`, `1on1-analysis`, `performance-narrative`).
**Decision:** Myna ships 24 feature skills and 6 steering skills, all as `SKILL.md` files under `agents/skills/myna-*/`. The install script copies them to `~/.claude/skills/myna-*/SKILL.md`. Steering skills are preloaded via the agent's `skills:` frontmatter (always in context). Feature skills are loaded on demand via Claude Code's progressive disclosure (only name and description in context until activated). No MCP server — vault operations use built-in tools. No steering inlining — they're proper skills with their own frontmatter. The agent body is lean: identity, skill directory table, routing logic for edge cases, and direct operations. Supersedes the skill count and MCP references in D049.
**Alternatives rejected:** Keep 14 consolidated skills (too broad — each skill prompt was very long and bundled unrelated workflows). Inline steering into agent body (steering skills are cleaner to maintain independently and avoid bloating the agent file). Keep the MCP server for vault operations (unnecessary complexity — Claude Code's built-in tools handle vault I/O directly, guided by a steering skill that teaches the patterns). Store skills as flat `.md` files instead of `SKILL.md` in directories (Claude Code's native mechanism expects `{skill-name}/SKILL.md` — using it directly avoids custom loading logic).

### D049 — Decoupled install: global Claude Code subagent via `claude --agent myna`
**Date:** 2026-04-11
**Context:** The install script from D047 generated a project-level `CLAUDE.md` at the repo root. This had two problems. First, it overwrote the developer-facing `CLAUDE.md` (project instructions for building Myna) with the runtime agent prompt (instructions for Claude as Myna Chief of Staff). The two files have different audiences but shared a file path, so running `install.sh` in the cloned repo clobbered the developer context. Second, it coupled runtime to repo location: the generated `CLAUDE.md` and the built MCP server both lived inside the cloned repo, so moving or deleting the repo broke the install. The user wanted a cleaner model where the repo is build-time only and all runtime artifacts live in stable, user-owned locations outside the repo. Claude Code's `--agent` feature (subagent-as-session via `claude --agent <name>`, verified to work) provides a natural fit: install writes a subagent file to `~/.claude/agents/myna.md` and the user invokes Myna by running `claude --agent myna` from any directory.

**Decision:** Install is fully decoupled from the cloned repo. Runtime artifacts live in two locations: (1) `~/.claude/agents/myna.md` is the Claude Code subagent definition, generated at install from `agents/main.md` plus the 5 steering files inlined into the body; (2) `~/.myna/` is Myna's own home containing `~/.myna/skills/` (14 plain markdown files copied from `agents/skills/`, read on demand by the agent), `~/.myna/mcp/myna-obsidian/` (built MCP server), `~/.myna/version` (installed version), and `~/.myna/install-manifest.json` (record of what was installed where). The MCP server is registered via `claude mcp add` pointing at `~/.myna/mcp/myna-obsidian/dist/index.js`, not the repo path. **The repo's `CLAUDE.md` is never touched by install.** Users invoke Myna with `claude --agent myna` from any working directory.

Subagent frontmatter is intentionally minimal — `name`, `description`, and `color` only. `model`, `tools`, `mcpServers`, `permissionMode`, `skills`, and `memory` are all omitted so Myna inherits session defaults; users can override any of these by editing `~/.claude/agents/myna.md` directly. The `skills:` frontmatter field is deliberately **not** used despite being available — it eagerly injects full skill content at session start, which for Myna's 14 skills would bloat context by several thousand lines per session even when most skills go unused. Instead, the agent body uses the existing lazy-load pattern: the skill directory table tells the agent to read `{{MYNA_HOME}}/skills/{skill}.md` when user intent matches, keeping startup context minimal and preserving `main.md`'s original design.

Steering rules are inlined into the agent body at install time because Claude Code's subagent spec recommends inlining for "always-active" rules. To keep the body size manageable, the 5 steering source files (safety, conventions, output, system, memory — ~560 lines combined) are tightened in a focused editing pass before inlining, targeting ~50% reduction by cutting human-facing rationale and examples while preserving every load-bearing rule. The source files remain the single source of truth — no separate compact version is maintained. Install is idempotent: re-running overwrites `~/.claude/agents/myna.md`, re-copies skills, and rebuilds the MCP, but never overwrites vault config files (`workspace.yaml`, `projects.yaml`, etc. — already handled by `create_config_if_missing`). A dedicated `update` command and an `--uninstall` flag are deferred to backlog (B019, B020). Supersedes D047.

**Alternatives rejected:** Copy agents to the Obsidian vault for a "self-contained" install (the vault is user data that Obsidian indexes and renders, not code; `node_modules` and compiled JS in a vault is a mess; updates would require a separate sync mechanism). Install to `~/.myna/` only and use a shell wrapper that `cd`s into it before launching `claude` (`claude --agent` is the Claude Code-native mechanism, doesn't require a PATH dependency, and aligns with D046 Claude-first design). Keep install in the repo and just document that the repo is the install location (doesn't fix the `CLAUDE.md` overwrite bug, still couples install to repo path). Use the `skills:` frontmatter field to preload all 14 skills (eager loading wastes context budget for skills that go unused in most sessions; lazy Read-on-demand pattern is what `main.md` was already designed for). Keep steering files as runtime-loaded separate files (spec recommends inline for always-active rules, runtime file resolution adds complexity with no benefit). Maintain two versions of steering files (full for humans, compact for runtime — drift risk; tightening the source files once preserves single source of truth). Rich frontmatter with `model`/`tools`/`permissionMode`/`memory` set upfront (KISS; adding fields is cheap, removing them once documented is harder).

### D048 — Memory model: 3-layer precedence with markdown learnings store *(superseded — see D048b)*
**Date:** 2026-04-11
**Context:** Myna had no place to capture experiential or procedural memory. Facts about specific entities (people, projects, meetings) live in their respective vault notes. But "user prefers terse drafts" or "avoid Friday afternoon meetings" — patterns and preferences emerging from interaction — had no home. `CLAUDE.md` captures user-authored bootstrap preferences but is not designed to evolve from interaction. Without an emergent-memory layer, every session restarts amnesiac to user preferences that aren't captured upfront. The brainstorming question was whether to build a dedicated memory MCP server (LanceDB, embeddings, hybrid full-text + vector search, composite scoring, auto-maintenance, S3 sync, ~12 tools) or to use plain markdown files behind the existing vault model.
**Decision (superseded — see D048b):** Memory was to be stored as plain markdown in per-domain files under `vault/_meta/` (email, meetings, tasks, people, general). Three behavioral-rule layers were to compose at runtime: (1) hard rules in steering skills always win; (2) active behavioral entries override `CLAUDE.md` because they reflect observed current state; (3) `CLAUDE.md` applies in the absence of a relevant entry. A dedicated skill was designed to capture, reflect, delete, and negotiate pushback on promoted entries. The skill would never touch `CLAUDE.md` — conflicts resolved by precedence at runtime. Entries would pass through Proposed → Active gating; observed patterns entered as Proposed and auto-promoted at observation count 3 across reflection passes. Reflection ran as the final wrap-up step. The `_meta/` folder was a new top-level vault folder for user-readable behavioral metadata, distinct from `_system/` internals.
**Alternatives rejected (for historical record):** Memory MCP server with LanceDB and embeddings (rejected as massive complexity; conflicts with KISS principle and local-first markdown promise; LLMs are excellent file readers and a single user's vault is not the corpus vector retrieval was designed for). Single flat behavioral-memory file (rejected — per-domain split allows selective loading and grows more naturally). Auto-editing `CLAUDE.md` when conflicts surface (rejected — `CLAUDE.md` is user-sacred; runtime precedence achieves alignment without touching the user's authored file). `CLAUDE.md` wins in all conflicts (rejected — forces manual edits every time preferences evolve). Reflection deferred to v1.1 (rejected — without reflection only explicit "remember that" statements get captured, which is too narrow for real use). Per-domain behavioral files in `_system/` (rejected — behavioral preferences are user-readable and user-editable, not internal plumbing).

### D048b — Memory model simplified to 2-layer; learn skill removed
**Date:** 2026-05-10
**Context:** D048 introduced a 3-layer memory model with a dedicated learn skill and a `_meta/` vault folder for per-domain behavioral files. During Phase 1 review, this was identified as unnecessary complexity: the extra vault folder adds surface area, the Proposed→Active promotion cycle is novel infrastructure to maintain, and Claude Code already provides a native memory mechanism (Claude Code memory, feedback type) that persists behavioral corrections across sessions without any extra skill or vault folder. The emergent-memory pattern was solving a real problem but with more mechanism than needed.
**Decision:** The memory model is simplified to 2 layers: (1) hard rules in steering skills always win; (2) `CLAUDE.md` / `workspace.yaml` applies otherwise. The learn skill is removed. Behavioral corrections observed during a session are saved to Claude Code memory (feedback type) at wrap-up — no separate vault folder, no promotion gating. The `_meta/` top-level vault folder is removed. Supersedes D048.
**Alternatives rejected:** Keep the learn skill with reduced scope (still adds vault folder and novel mechanism; native Claude Code memory is simpler and already built). Keep a plain behavioral-memory log in `_meta/` without the learn skill (orphaned file with no write path — confusing). Keep session reflection at wrap-up as a standalone step without the learn skill (reflection still needs to write somewhere; native memory is the right target).

### D047 — Phase 2 (Install) simplified to shell script for Claude Code
**Date:** 2026-04-07
**Context:** With Claude Code as the v1 runtime (D045), the install step no longer needs to "package" agent artifacts into a tool-specific format. Claude Code reads markdown files directly and uses CLAUDE.md as its entry point. The adapter-layer concept from D038 — where the installer would translate content into Kiro CLI's expected formats — is unnecessary. The install step is now much simpler.
**Decision:** Phase 2 (Install) is a shell script that: (1) generates a project-level CLAUDE.md that loads main.md, steering files, and provides the skill directory for Claude Code, (2) registers the myna-obsidian MCP server with Claude Code via `claude mcp add`, (3) creates the `myna/` vault folder structure in the user's Obsidian vault, (4) copies `.example` config files to `_system/config/`. No adapter packaging, no format translation, no tool-specific syntax generation. The install script remains user-involved (not autonomous) per the original D035 intent. Supersedes Phase 2 scope from D035.
**Alternatives rejected:** Keep the adapter-packaging scope (no adapter needed — Claude Code reads markdown directly). Fully automated install with no user involvement (user wants direct input on install experience). No install script at all (manual setup is too error-prone — users need vault structure, MCP registration, and CLAUDE.md generation automated).

### D046 — Claude-first design; relax tool-agnostic content/adapter separation
**Date:** 2026-04-07
**Context:** D038 established a strict two-layer architecture: a tool-neutral content layer (no tool-specific syntax) and a tool-specific adapter layer (install-time packaging). This was well-intentioned — it aimed to make adding future AI tool support cheap. In practice: (a) we never built adapters for other tools, (b) the portability constraint added complexity to every design decision, (c) the content files are plain markdown and YAML, which are inherently readable by any LLM — portability comes from the file format itself, not from an architectural abstraction. The two-layer separation was speculative infrastructure for a future that may never arrive.
**Decision:** The strict two-layer separation from D038 is relaxed. Agent instructions can reference Claude Code capabilities directly (e.g., how CLAUDE.md loads, how MCP servers are configured, how skills are activated). The "portability test" ("would this file need changes to work on a different AI tool?") is removed as a design constraint. Content remains plain markdown and YAML — inherently portable. If someone wants to run Myna on Gemini or Codex in the future, they can read the markdown files and write their own wiring. That's an open-source community contribution, not something we architect for upfront. D002 (AI model agnostic) is preserved in spirit: the content is LLM-readable, not Claude-proprietary. Supersedes D038 and the adapter-related parts of D007.
**Alternatives rejected:** Keep the strict two-layer separation (speculative complexity — we've never needed it and it taxed every design decision). Go fully Claude-proprietary with Claude-specific XML tags or syntax in content files (unnecessarily locks content to Claude; plain markdown is portable by nature). Remove D002 entirely (the principle of not being proprietary is still valuable — we just implement it through file format choice rather than architectural abstraction).

### D045 — Switch from Kiro CLI to Claude Code as v1 runtime
**Date:** 2026-04-07
**Context:** Myna was designed with Kiro CLI as the v1 target runtime (D035). During Phase 1 (Build), it became clear that Claude Code is a better fit: (a) Claude Code has native MCP support — Myna's myna-obsidian MCP server works directly via `claude mcp add`, (b) Claude Code uses CLAUDE.md as its entry point, which maps naturally to Myna's main agent instructions, (c) Claude Code has a large and growing user base, (d) the project is already being built with Claude Code, so the team has deep familiarity with its capabilities and constraints.
**Decision:** Claude Code replaces Kiro CLI as the v1 runtime target. All Phase 2 (Install) and Phase 3 (Ship) work targets Claude Code. The install script generates a CLAUDE.md for the user's Myna project directory, registers the MCP server with Claude Code, and creates the vault structure. Kiro CLI, Gemini, Codex, and other AI tools are potential post-launch community contributions, not v1 scope. Supersedes D035.
**Alternatives rejected:** Keep Kiro CLI as v1 target (less natural fit — Kiro CLI has its own opinionated structure that requires adapter work; the team has less experience with it). Target both Claude Code and Kiro CLI in v1 (scope creep — each tool has different install mechanics and runtime behavior). Target a generic "any tool" install (the adapter-layer approach from D038, which proved to be speculative overhead).
